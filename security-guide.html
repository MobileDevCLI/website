1. Core Security Principles

The Cardinal Rules

  1. Never commit secrets to git - API keys, passwords, tokens
  2. Use aliases, not real names - Protect your identity
  3. Always use .gitignore - Prevent accidental exposure
  4. Rotate exposed credentials immediately - Assume compromise

The Golden Rule

If a credential is committed to git, even briefly, consider it compromised. Git history is permanent. Even deleted files remain in the repository history.

2. Protecting Secrets

Never Commit These

Type Pattern Example
API Keys sk-*, pk_*, api_key sk-abc123...
Passwords password, secret DB_PASSWORD=...
JWT Secrets eyJ... (long) Server signing keys
Private Keys *.pem, *.key SSH keys, SSL certs

Safe to Commit (Public by Design)

  • Supabase ANON key - Designed for client-side use
  • Public OAuth client IDs - Not secret
  • Public API URLs - Endpoints, not credentials

Use Environment Variables

# .env file (gitignored)
API_SECRET_KEY=sk-your-secret-key
DATABASE_PASSWORD=your-password

# In code
const key = process.env.API_SECRET_KEY;

Check Before Committing

# Search for potential secrets
grep -rn "api_key\|secret\|password\|token\|sk-" .

# Check staged changes
git diff --staged

3. Personal Data Protection

Use Aliases

Instead of your real name, use a consistent alias:

  • Good: "Samblamz", "DevUser123"
  • Bad: "John Smith", your real name

Never Expose

Keep Private

  • Real names (yours, family, friends)
  • Physical addresses
  • Phone numbers
  • Personal email addresses
  • Location data (GPS coordinates, city names)
  • Relationship details

Business vs Personal

Create separate identities:

  • Business email: contact@yourdomain.com
  • Business alias: Your brand/company name
  • Personal: Keep completely separate

4. Git Security

Essential .gitignore

# Personal/private files
PERSONAL_*.md
PRIVATE_*.md
*_SECRET.*

# Environment files
.env
.env.local
.env.*.local

# Credentials
*.pem
*.key
credentials.json
secrets.json

# Dependencies
node_modules/

# OS files
.DS_Store
Thumbs.db

If You Commit a Secret

  1. IMMEDIATELY rotate the credential - Get new API key
  2. Check for unauthorized access
  3. Update all systems using that credential
  4. Consider using git-filter-repo to remove from history

Warning: Git History is Permanent

Even after deleting a file, it remains in git history. Removing from history requires rewriting it, which affects all collaborators. Prevention is better than cure.

5. Supabase Security

Key Types

Key Type Safe in Client? Use Case
ANON Key Yes Browser/mobile apps
SERVICE Key NEVER Server-side only

Row Level Security (RLS)

Always enable RLS on tables with user data:

-- Enable RLS
ALTER TABLE users ENABLE ROW LEVEL SECURITY;

-- Users can only see their own data
CREATE POLICY "Users see own data"
ON users FOR SELECT
USING (auth.uid() = user_id);

Why ANON Key is Safe

The ANON key is designed to be public. Security is enforced through:

  • Row Level Security policies
  • Authentication requirements
  • Server-side validation

6. Claude Code Security

Permission Modes

Mode Safety Use Case
Default Safest Asks before actions
--permission-mode=plan Moderate Pre-approved commands
Full Autonomous Mode Risky Sandboxed only

Safe Commands to Pre-Allow

// .claude/settings.json
{
  "allowedCommands": [
    "Bash(npm install:*)",
    "Bash(npm run build:*)",
    "Bash(git add:*)",
    "Bash(git commit:*)",
    "Bash(ls:*)"
  ]
}

Never Pre-Allow

  • rm -rf - Can delete everything
  • sudo - System-level access
  • curl with credentials
  • Commands accessing /etc/ or system files

7. Mobile/Terminal Security

Terminal Best Practices

  • Storage access: Only grant when needed
  • Wake lock: Use for long sessions
  • tmux: Session persistence without exposure

Screenshot Awareness

Screenshots May Contain Secrets

Be careful when sharing screenshots. They may contain:

  • Terminal output with credentials
  • API keys in code
  • Personal information
  • File paths revealing usernames

Secure Storage

# Don't store secrets in plain text
# BAD:
echo "API_KEY=sk-secret" > config.txt

# GOOD: Use environment variables
export API_KEY="sk-secret"  # Session only

8. Security Checklist

Before Every Commit

  • Search for API keys and secrets
  • Check for personal names/addresses
  • Verify .gitignore includes sensitive files
  • Review git diff --staged

Project Setup

  • Create comprehensive .gitignore
  • Use environment variables for secrets
  • Set up alias instead of real name
  • Enable RLS on Supabase tables

Regular Audits

  • Scan for exposed credentials
  • Review git history for sensitive data
  • Check public repos for personal info
  • Rotate old credentials

Quick Security Scan

# Run this before pushing
grep -rn "api_key\|secret\|password\|sk-\|pk_" . \
  --include="*.js" --include="*.html" --include="*.json"

# Check for personal info
grep -rn "@gmail\|@yahoo" . --include="*.html"

Conclusion

Security is about prevention, not reaction. By following these practices:

  1. Separate public and private - Use .gitignore religiously
  2. Use aliases - Protect your real identity
  3. Never commit secrets - Use environment variables
  4. Audit regularly - Check before pushing
  5. Assume compromise - Rotate exposed credentials immediately

This security framework was developed through real-world experience building MobileCLI. Every lesson learned is captured here so you can build securely from day one.
