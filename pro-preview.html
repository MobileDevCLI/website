5-Minute Setup

Get from zero to fully autonomous AI development in 5 minutes. This is the fast path — the Complete Guide tab has the deep explanations.

1 Install Termux (from F-Droid)

Critical: The Google Play Store version is outdated and broken. You MUST use F-Droid.

Download Link 
https://f-droid.org/packages/com.termux/

Also install Termux:API from F-Droid for hardware access (camera, notifications, clipboard, etc.).

2 Update Packages

Open Termux and run:

Bash 
pkg update -y && pkg upgrade -y
3 Install Core Packages
Bash 
pkg install -y nodejs git gh openssh termux-api
4 Install Claude Code
Bash 
npm install -g @anthropic-ai/claude-code
5 Authenticate
Bash 
# Authenticate with Claude (opens browser)
claude

# Authenticate with GitHub
gh auth login
6 Enable Autonomous Mode

This is the secret sauce. Autonomous mode lets Claude work without asking for permission on every action.

Bash — THE POWER COMMAND 
claude --dangerously-skip-permissions
Use Responsibly

This flag gives Claude full system access. Only use in sandboxed environments or when you trust what you're building.

7 Keep It Running Forever
Bash — Prevent Android from Killing Termux 
# Acquire wake lock (keeps CPU active)
termux-wake-lock

# Install tmux for persistent sessions
pkg install -y tmux

# Start a named session
tmux new -s dev

# Now run Claude inside tmux
claude --dangerously-skip-permissions
You're Done

That's it. You now have a fully autonomous AI development environment running on your phone. Claude can read files, write code, run commands, and deploy to production — all while you sleep.

Understanding the Architecture

Before diving into commands, it's essential to understand what we're building and why it works. This section explains the underlying architecture that makes AI-powered mobile development possible.

The Stack

The MobileCLI development environment consists of four layers:

Layer Component Purpose
Hardware Android Phone (ARM64) Physical compute device. Any Android 7.0+ with 4GB+ RAM.
Linux Environment Termux Full Linux terminal emulator. Provides bash, package management, filesystem access.
Runtime Node.js + Git + GitHub CLI JavaScript runtime, version control, GitHub integration.
AI Interface Claude Code AI CLI that reads, writes, and executes. The brain of the operation.

Why This Works

The key insight is that the command line is the great equalizer. On CLI, a $200 Android phone has functionally the same capabilities as a $3000 MacBook Pro. You have:

  • Full filesystem access (read/write any file)
  • Process execution (run any command)
  • Network access (git push, API calls, deployments)
  • Package management (install any npm/apt package)

The AI doesn't care about screen size or touch vs keyboard. It operates through the terminal — and the terminal works identically on every platform.

Termux Deep Dive

Termux is an Android terminal emulator and Linux environment app. It provides a full Linux system without requiring root access.

Why F-Droid, Not Play Store?

The Play Store version of Termux was abandoned in 2020 due to Google's API restrictions. It's broken on modern Android. The F-Droid version is actively maintained and works correctly.

Download Links

The CLI receives three OAuth grants:

  • user:inference — Run Claude models
  • user:profile — Access your account info
  • user:sessions:claude_code — Manage Claude Code sessions

Autonomous Mode Explained

By default, Claude Code asks for permission before every file write, command execution, or significant action. This is safe but slow.

Autonomous Mode (the --dangerously-skip-permissions flag) removes all permission checks. Claude acts immediately without confirmation.

Why It's Called "Dangerous"

With this flag, Claude can:

  • Delete any file
  • Execute any shell command
  • Push to any git remote
  • Make API calls
  • Modify system configuration

In a sandboxed environment (like Termux on your phone), this is fine. On a production server with real data, be careful.

Permission Modes

Bash — Different Permission Levels 
# Full autonomous (no prompts at all)
claude --dangerously-skip-permissions

# Alternative: permission mode that doesn't ask
claude --permission-mode=dontAsk

# Default: asks for each action
claude

Session Persistence

Android aggressively kills background apps to save battery. Without proper configuration, Termux will be killed when you lock your phone.

Wake Lock

The wake lock keeps the CPU active even when the screen is off:

Bash 
# Acquire wake lock (shows notification)
termux-wake-lock

# Release wake lock
termux-wake-unlock

tmux for Session Survival

tmux is a terminal multiplexer. It keeps sessions alive even if the terminal window closes.

Bash — tmux Commands 
# Install tmux
pkg install -y tmux

# Create a new named session
tmux new -s dev

# Detach from session (keeps it running)
# Press: Ctrl+b, then d

# List running sessions
tmux ls

# Reattach to a session
tmux attach -t dev

# Kill a session
tmux kill-session -t dev

Disable Battery Optimization

Go to Android Settings → Apps → Termux → Battery → Set to "Unrestricted" or "Don't optimize".

CLAUDE.md — Knowledge Compounding

This is perhaps the most powerful technique in AI-assisted development. CLAUDE.md is a file that Claude reads at the start of every session.

How It Works

  1. Create a file called CLAUDE.md in your project root
  2. Write project context, rules, and preferences
  3. Every time Claude makes a mistake, add a rule to prevent it
  4. Over time, the file becomes a highly tuned instruction set
  5. Claude reads this at session start and "knows" all your preferences

Template

CLAUDE.md Template 
# Project Instructions

## Quick Context
This is [project name]. We're building [description].
Owner: [your alias]
Website: [url]
GitHub: [org/repo]

## Tech Stack
- Vanilla HTML/CSS/JS (no frameworks)
- Three.js for 3D graphics
- Supabase for auth and database
- Vercel for hosting

## Important Rules
1. Never share personal info — user is anonymous
2. Always push to BOTH repos: origin AND website
3. Keep repos PUBLIC (Vercel needs access)
4. Mobile-first — always test on phone
5. Create tags before major changes

## Code Style
- Prefer `type` over `interface` (TypeScript)
- Never use `enum` — use string literal unions
- Always handle errors explicitly
- Use absolute imports, not relative

## Common Commands
```bash
# Quick deploy
git add -A && git commit -m "message" && git push origin master && git push website master

# Run tests
npm run test

# Build
npm run build
```

## Learned Lessons
- Don't add console.log to production code
- The Play Store Termux is broken — use F-Droid
- vercel.json needs cache headers to prevent stale content
- Always check both repos are synced before pushing
Pro Tip

The "Learned Lessons" section is where the magic happens. Every time Claude does something wrong, add it here. The knowledge compounds across sessions.

Pro Workflow Patterns

These advanced patterns maximize productivity for AI-assisted development. These techniques come from the creator of Claude Code and are used to build production applications.

Parallel Sessions: Run 5-10 Claudes Simultaneously

The key insight: don't wait. While one Claude is thinking or executing, prompt another.

Terminal Sessions

  • Run 5 Claudes in parallel in terminal tabs numbered 1-5
  • Use system notifications to know when Claude needs input
  • Configure terminal notifications in Termux settings

Web Sessions

  • Run 5-10 additional Claudes on claude.ai/code
  • Hand off local sessions to web using & (background process)
  • Use --teleport to transfer sessions between terminal and web
  • Start sessions from phone (Claude iOS/Android app) and check later
Bash — Parallel Session Management 
# Background a session
claude &

# Transfer session to web
claude --teleport

# Check running Claude processes
ps aux | grep claude
Why This Works

Different Claudes can work on different aspects of the same project. While one builds a feature, another fixes bugs, another writes documentation. You multiply your output.

Model Selection: Always Use Opus 4.5 with Thinking

Use the best model available. Bigger is actually faster in practice.

Why Opus Wins

  • Less steering required — Opus understands context better
  • Better at tool use — Fewer errors, less back-and-forth
  • Almost always faster end-to-end despite being "slower" per token

The math: If Sonnet needs 5 back-and-forth corrections but Opus gets it right first try, Opus wins every time. The total time to completion is what matters, not tokens-per-second.

Bash — Specify Model 
# Use Opus 4.5 (recommended)
claude --model opus

# With thinking enabled
claude --model opus --thinking
Plan Mode First: Think Before Coding

Most sessions should start in Plan mode. Don't jump straight to implementation.

The Workflow

  1. Enter Plan mode: shift+tab twice
  2. Describe what you want to build
  3. Go back and forth with Claude until the plan is solid
  4. Switch to auto-accept edits mode
  5. Claude usually 1-shots the implementation from a good plan
Why Planning First

A good plan is the foundation for quality output. You catch architectural issues before code is written. It's much faster than fixing bad code after the fact.

PostToolUse Hooks: Auto-Format Everything

Run commands automatically after Claude edits files.

.claude/settings.json 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "npm run format || true"
      }
    ]
  }
}

Claude generates well-formatted code 90% of the time. The hook handles the last 10%. The || true prevents formatting errors from blocking work.

Verification: The 2-3x Quality Multiplier

This is the most important insight for quality output.

If Claude has a feedback loop — a way to verify its work — it will 2-3x the quality of the final result. The key is giving Claude a way to check if what it built actually works.

Domain Verification Method
Web UI Chrome extension — opens browser, tests UI, iterates
CLI Tools Run bash commands, check exit codes
Libraries Run test suite after changes
Mobile Apps Phone simulator or device testing
Games Serve locally, screenshot, visual check
Key Insight

"Verification looks different for each domain. It might be as simple as running a bash command, or running a test suite, or testing the app in a browser or phone simulator. Make sure to invest in making this rock-solid. It's the highest-leverage improvement you can make."

Technique

The 2-3x Quality Multiplier

When Claude has a feedback loop — a way to verify its work — output quality increases 2-3x. This is the single most important technique.

The loop works like this:

  1. Claude writes code
  2. Claude runs/tests the code
  3. Claude sees the result (pass/fail, screenshot, output)
  4. Claude iterates until correct

Without verification, Claude guesses and moves on. With verification, Claude knows when it's done.

Debugging

Multi-Layer Cache Debugging

We spent hours debugging why site changes weren't showing. The issue: four caching layers.

  1. Browser Cache — Your browser stores pages locally
  2. Vercel Edge Cache — CDN caches at edge nodes
  3. ISP/DNS Cache — Internet provider caching
  4. OS/Network Cache — Device-level caching
Bash — Bypass All Caches 
# This bypasses ALL caching layers
curl -sL "https://www.yoursite.com" | grep "unique text"

# If curl shows new content but browser doesn't → cache problem

Solution: Add to vercel.json:

vercel.json 
{
  "headers": [
    {
      "source": "/(.*).html",
      "headers": [
        {
          "key": "Cache-Control",
          "value": "public, max-age=0, must-revalidate"
        }
      ]
    }
  ]
}

Troubleshooting Guide

Solutions to common problems encountered during setup and development.

Claude Code Not Installing?

Symptom: npm install -g @anthropic-ai/claude-code fails.

Solutions:

  1. Make sure Node.js is installed: pkg install nodejs
  2. Try with sudo: Not needed in Termux (you're already root-equivalent)
  3. Clear npm cache: npm cache clean --force
  4. Check npm version: npm --version (should be 8+)
  5. Try reinstalling Node: pkg reinstall nodejs
Session Killed by Android?

Symptom: Termux closes when phone locks or after some time.

Solutions:

  1. Run termux-wake-lock before starting work
  2. Use tmux: tmux new -s dev
  3. Disable battery optimization for Termux in Android Settings
  4. Enable "Acquire Wakelock" notification in Termux settings
  5. Keep the phone plugged in for long sessions
Storage Access Issues?

Symptom: Can't access ~/storage/ or files in Downloads/Screenshots.

Solutions:

  1. Run termux-setup-storage (only needed once)
  2. Accept the permission dialog that appears
  3. If dialog doesn't appear, check Android Settings → Apps → Termux → Permissions → Storage
  4. After granting, restart Termux
GitHub Authentication Failed?

Symptom: gh auth login fails or git push asks for password.

Solutions:

  1. Run gh auth login and follow prompts
  2. Choose HTTPS (not SSH) if you don't have SSH keys set up
  3. Make sure you're logged into github.com in your browser
  4. Check auth status: gh auth status
  5. If using SSH, make sure openssh is installed: pkg install openssh
tmux Session Lost?

Symptom: Can't reattach to tmux session.

Solutions:

  1. List sessions: tmux ls
  2. Attach to session: tmux attach -t dev
  3. If no sessions, Termux was killed — restart with tmux new -s dev
  4. Make sure wake lock is active: termux-wake-lock
Vercel Not Deploying?

Symptom: Push succeeds but site doesn't update.

Diagnosis:

Bash — Test What Server Has 
curl -sL "https://yoursite.com" | grep "unique text"

If curl shows new content but browser doesn't: Browser cache. Clear it or use incognito.

If curl shows old content:

  1. Check you pushed to the right repo (the one Vercel watches)
  2. Vercel Dashboard → Project → Settings → Git
  3. Check "Connected Git Repository" is correct
  4. Click Disconnect, then reconnect
  5. Manual redeploy: Deployments → ... → Redeploy (uncheck cache)
